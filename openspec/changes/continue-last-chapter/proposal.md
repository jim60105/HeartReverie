## Why

LLM responses are frequently truncated mid-sentence, mid-paragraph, or before the requested 20-line minimum is reached — caused by upstream timeouts, provider stream errors (now surfaced as `ChatError("llm-stream", …)` per the recent `streaming-cancellation` fix), accidental abort by the user, or simply the model deciding to stop early. Today the only recovery options are **edit** the chapter (manual stitching) or **rewind + resend** (which discards the partial output and pays the upstream cost again). Both are wasteful and break the user's flow.

A "Continue" affordance lets the user resume generation in place: chapters 1..n−1 are sent as the regular history, chapter n's existing content is appended as the **prefix of the assistant turn**, and the LLM continues writing from exactly where it stopped — without restating, without creating chapter n+1, and without losing the bytes already on disk.

## What Changes

- Add a new backend operation that resumes an existing chapter. The path is a new HTTP endpoint and a new WebSocket message type rather than overloading `chat:send`, because the request body has no `message` field (the model's prefix replaces user input as the trigger) and because the persistence semantics differ (append in place vs. open-and-truncate a fresh chapter file).
- HTTP: `POST /api/stories/:series/:name/chat/continue`. No body fields are required. Path-traversal-safe (re-uses `validateParams` + `safePath`). Rate-limited by the existing per-passphrase chat rate limiter that already guards `POST /api/stories/:series/:name/chat`.
- WebSocket: `{ type: "chat:continue", id, series, story }`. The server replies with the same `chat:delta` / `chat:done` / `chat:error` / `chat:aborted` envelopes already defined for `chat:send`, so the streaming, abort, and idle-timer code paths are reused unchanged. `id` is correlation-only (client-generated).
- Add a new `WriteMode` variant `{ kind: "continue-last-chapter"; targetChapterNumber: number; existingContent: string }` to `streamLlmAndPersist()` in `writer/lib/chat-shared.ts`. In this mode the helper SHALL: (a) NOT open the chapter file with `truncate: true`; (b) open the file with `append: true` so streamed deltas are written to the tail of the existing bytes; (c) skip the `pre-write` hook (no new user message, no leading wrapper); (d) still dispatch `response-stream` for each delta and `post-response` after success with `source: "continue"`; (e) reuse the existing reasoning/`<think>` framing and OpenRouter usage / mid-stream-error / abort logic without duplication.
- Add a new `executeContinue()` function in `writer/lib/chat-shared.ts`, parallel to `executeChat()`, that: (a) refuses when the story has zero chapter files (`ChatError("no-chapter", …, 400)`); (b) **re-reads the latest chapter's raw bytes from disk on every invocation** (no caching across calls) so any manual edits the user has made to the file — to the `<user_message>` block, to the LLM-generated prose, or both — are picked up verbatim; (c) parses the freshly-read chapter into two parts via `parseChapterForContinue()` (a new helper in `writer/lib/story.ts`): `userMessageText` = the *content* of the **first** `<user_message>…</user_message>` block (with the wrapping tags removed) when present, and `assistantPrefill` = the chapter bytes with that first user_message block removed and then run through `stripPromptTags()` to drop any other plugin framing tags; (d) refuses when *both* `userMessageText.trim() === ""` *and* `assistantPrefill.trim() === ""` (`ChatError("no-content", "Latest chapter is empty; nothing to continue", 400)`) — at least one of the two parts must carry semantic content; (e) writes the **unstripped, original on-disk** chapter bytes as the in-place file prefix (the persistence layer does NOT touch existing bytes); (f) builds the prompt via a new `buildContinuePromptFromStory()` that sends chapters 1..n−1 in `previous_context` (existing behaviour for that array) and renders `system.md` with `userInput: userMessageText` so the trailing user turn carries the user's (possibly edited) intent, then appends `{ role: "assistant", content: assistantPrefill }` as the final message; (g) acquires the same per-story generation lock as `executeChat()` via `tryMarkGenerationActive` / `clearGenerationActive`.
- Extend the rendered messages array. The Vento template `system.md` does NOT need to be modified — the continue path renders the template with `userInput: userMessageText` (the user's prompt extracted live from chapter n on disk, so post-edit values are honoured) and then appends one extra `{ role: "assistant", content: assistantPrefill }` to the message list before handing it to the upstream `fetch()`. The `system.md` template's existing trailing user turn carries `userMessageText` exactly as a normal `chat:send` would carry the user's typed input; the appended assistant message is the prefill. This relies on OpenAI's chat-completions contract: when the **last** message in `messages` is `role: "assistant"`, most providers (OpenAI, OpenRouter, DeepSeek, Anthropic via OpenRouter passthrough) treat it as a "prefill" and continue from its `content` without restating it. **Caveat (must be documented in design.md):** strict OpenAI does NOT officially document prefill via the public `chat/completions` API and may inject a leading newline or refuse with some models; in practice the project ships with `LLM_API_URL` defaulting to OpenRouter, where prefill is documented and reliable. We accept this provider-dependence as a known limitation and surface it via release notes; no `prefill` / `continue` request flag is added (no provider in the project's supported set requires one — OpenRouter infers it from the trailing assistant message). When `userMessageText` is empty (chapter has no `<user_message>` block — e.g. user deleted it manually) the rendered user turn carries `""` and the model sees only the assistant prefill; when `assistantPrefill` is empty (user_message block exists but no prose has been generated yet) the trailing assistant message has empty content and the model effectively re-runs the user turn from scratch, equivalent to a re-issue of `chat:send`.
- Frontend: add a `Continue` (繼續) action button rendered next to the Send/Stop/Resend buttons inside `reader-src/src/components/ChatInput.vue`. The button SHALL be visible whenever a story is selected and SHALL be disabled when (a) `isLoading` is true (active generation), (b) the loaded chapter list is empty, or (c) the last chapter's content is empty / whitespace-only. The button SHALL invoke a new `continueLastChapter(series, story)` method on `useChatApi()` which dispatches `chat:continue` over WebSocket (when connected/authenticated) or POSTs to the new HTTP endpoint, and SHALL update the existing `isLoading` / `streamingContent` / `errorMessage` / token-usage state in exactly the same way `sendMessage()` does.
- Add a new `chat:send:before` analogue is **not** dispatched for continue (continue has no user-visible message to mutate). The `response-stream` and `post-response` plugin hooks ARE dispatched, with the existing `correlationId` / `chapterPath` / `chapterNumber` / `series` / `name` / `storyDir` fields plus a new `source: "continue"` discriminator on `post-response` (existing values: `"chat"`, `"plugin-action"`).
- Append-mode persistence MUST NOT corrupt the existing `<user_message>…</user_message>` block written at the head of the chapter by the original `chat:send` round. The continue write path opens the file with `{ write: true, append: true, mode: 0o664 }` (no `truncate`, no `create` — the file is required to exist) so streamed deltas are concatenated to the existing tail. The `response-stream` hook chunk transformation continues to apply per chunk.
- Token-usage tracking: when the upstream emits `usage`, append a record to `_usage.json` with `chapter: targetChapterNumber` and `model: llmConfig.model`, exactly as `executeChat()` does. The frontend `useUsage` composable's existing `pushRecord` flow handles UI refresh.
- Tests:
  - `tests/writer/lib/chat_shared_continue_test.ts`: unit tests for `executeContinue()` covering happy-path append, refusal on zero chapters, refusal on empty-latest-chapter, abort during continue, mid-stream-error during continue, and that the upstream request body's last message has `role: "assistant"` with content equal to the stripped chapter-n prefix.
  - `tests/writer/routes/chat_test.ts`: extend with HTTP `POST /chat/continue` happy path + 400 (no chapter) + 400 (empty chapter) + 499 (abort) + 502 (llm-stream).
  - `tests/writer/routes/ws_test.ts`: extend with WS `chat:continue` happy path + abort during continue.
  - `tests/reader-src/components/ChatInput.test.ts` (or the closest existing front-end test file): assert the Continue button's enabled/disabled gating against the four conditions.
- Documentation:
  - Add the `continue-last-chapter` capability spec describing the user-facing behaviour, prefill mechanism, persistence guarantees, and the OpenAI-compatible-provider caveat.
  - Modify the `chat-input` capability spec to document the Continue button and its enable/disable gates.
  - Modify the `websocket-chat-streaming` capability spec to add the `chat:continue` envelope and reaffirm the shared abort / idle-timer / error semantics.
  - Modify the `writer-backend` capability spec to add the `POST /chat/continue` HTTP endpoint scenario.

## Capabilities

### New Capabilities

- `continue-last-chapter`: the contract for resuming generation on the latest chapter — what triggers it (HTTP `POST /chat/continue`, WebSocket `chat:continue`), how the prompt is constructed (chapters 1..n−1 as `previous_context`, chapter n stripped and injected as a trailing `assistant` message that the provider treats as a prefill), how persistence appends in place without truncation, and how the existing streaming / abort / token-usage / mid-stream-error pipelines are reused.

### Modified Capabilities

- `chat-input`: gains the Continue button requirement (visibility, enable/disable gating against `isLoading` / chapter count / empty-latest-chapter, dispatch via `useChatApi.continueLastChapter()`).
- `websocket-chat-streaming`: gains the `chat:continue` envelope and a scenario asserting that abort and connection-close cleanup work identically to `chat:send`.
- `writer-backend`: gains the `POST /api/stories/:series/:name/chat/continue` HTTP endpoint scenario alongside the existing `POST /chat` / `chat:send` proxy requirements.

## Impact

- **Code**:
  - `writer/lib/chat-shared.ts` — new `WriteMode` variant `continue-last-chapter`, new `executeContinue()` function, updated `streamLlmAndPersist()` switch arms, new file-open call with `{ append: true }`.
  - `writer/lib/story.ts` — new `buildContinuePromptFromStory()` and a new `parseChapterForContinue()` helper that splits a chapter's on-disk bytes into `{ userMessageText, assistantPrefill }` (extracts the first `<user_message>…</user_message>` block as the user turn, removes that block, then runs `stripPromptTags()` on the remainder for the assistant prefill). `buildContinuePromptFromStory()` re-reads the latest chapter from disk on every call (no in-memory cache) so user edits are honoured.
  - `writer/routes/chat.ts` — register `POST /:series/:name/chat/continue`.
  - `writer/routes/ws.ts` — handle `chat:continue` in the type switch; reuse the existing per-id `AbortController` map and idle-timer cleanup.
  - `writer/types.ts` — extend `BuildPromptFn` (or add `BuildContinuePromptFn`) and the `WriteMode` discriminated union if exported.
  - `reader-src/src/components/ChatInput.vue` — render the Continue button + emit/handle the `continue` event.
  - `reader-src/src/composables/useChatApi.ts` — new `continueLastChapter(series, story)` method covering both WS and HTTP paths.
  - `reader-src/src/composables/useChapterNav.ts` (or wherever the chapter list is exposed) — surface `chapterCount` and `latestChapterIsEmpty` reactive refs the button can gate on.
- **Tests**: new `tests/writer/lib/chat_shared_continue_test.ts`, plus extensions to `tests/writer/routes/chat_test.ts`, `tests/writer/routes/ws_test.ts`, and the front-end test for `ChatInput.vue`.
- **APIs**: one new HTTP endpoint (`POST /api/stories/:series/:name/chat/continue`), one new WebSocket message type (`chat:continue`). Existing `chat:delta`, `chat:done`, `chat:error`, `chat:aborted` envelopes are reused without schema change. RFC 9457 Problem Details map: existing `no-chapter` → 400 and `no-content` → 502 codes are reused; `no-content` is repurposed for the empty-chapter case at HTTP 400 specifically when surfaced from the continue route — no new `ChatError.code` literal is added.
- **LLM contract**: the upstream request now sometimes ends with an `assistant` message instead of a `user` message. This is documented OpenAI-compatible behaviour for OpenRouter (prefill); strict OpenAI may insert a leading newline or behave inconsistently with some models. The project's default `LLM_API_URL` (OpenRouter) supports it reliably; deployments swapping in stricter providers via `.env` accept the limitation.
- **Logs**: the LLM interaction log gains entries with `mode: "continue-last-chapter"` (mirroring the existing `mode: "write-new-chapter"` discriminator already logged by `streamLlmAndPersist`). No new log file or schema.
- **Plugins / hooks**: `pre-write` is NOT dispatched for the continue path (no new user message, no fresh wrapper). `response-stream` and `post-response` ARE dispatched; `post-response.source` gains a new value `"continue"` alongside the existing `"chat"` and `"plugin-action"`. Plugins that branch on `source` need to handle the new value (the audit-logging, response-notify, and context-compaction plugins all already use a default branch and need no change).
- **Specs**: one new capability (`continue-last-chapter`), three MODIFIED capabilities (`chat-input`, `websocket-chat-streaming`, `writer-backend`).
- **Backward compatibility / migration**: none required. The project is pre-release with zero deployed users (per `AGENTS.md` and the `streaming-cancellation` archived change); we are free to ship the new HTTP route and WS envelope without a deprecation window.
- **Dependencies / runtime / config**: none. No new env vars, no new packages.
