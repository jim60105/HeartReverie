## ADDED Requirements

### Requirement: Continue endpoint resumes generation in place

The server SHALL expose `POST /api/stories/:series/:name/chat/continue` as a sibling route to the existing `POST /api/stories/:series/:name/chat`. The endpoint SHALL:

1. Apply the existing `validateParams` middleware for path-traversal safety on `:series` and `:name`.
2. Apply the existing per-passphrase rate limiter that already guards `POST /chat`.
3. Require no JSON body fields. The endpoint SHALL accept (and ignore) any JSON body, including empty (`{}`) or absent.
4. Invoke `executeContinue()` from `writer/lib/chat-shared.ts` (see the `continue-last-chapter` capability for the full contract), passing `c.req.raw.signal` as the abort signal so that HTTP client disconnect propagates to the upstream LLM fetch.
5. Stream the response identically to `POST /chat`: the chapter file SHALL receive each delta in real time (appended to the existing tail), and the HTTP response body SHALL be the JSON `{ chapter, content, usage }` envelope used by the existing chat route. The response `content` field SHALL be **the full updated chapter** — semantically equal to the original pre-continue chapter bytes concatenated with all bytes appended during this stream (post-transform, e.g. after `<think>` framing). Implementations SHALL produce this value by reading `chapterContentAfter` from the `streamLlmAndPersist()` return value (which is itself populated by re-reading the chapter file from disk after the stream completes), and SHALL NOT use the `content` field of that return value (which contains only the newly generated bytes by the existing `streamLlmAndPersist` contract). `chapter` is the chapter number that was extended.
6. Translate `executeContinue()` errors to HTTP responses identically to `POST /chat`:
   - `ChatAbortError` → HTTP 499 RFC 9457 Problem Details `{ title: "Client Closed Request", status: 499, detail: "Generation aborted by client" }`.
   - `ChatError` → HTTP `httpStatus` with Problem Details `{ title: ERROR_TITLES[code] | "Internal Server Error", status: httpStatus, detail: <ChatError.message> }`. The existing `ERROR_TITLES` map already covers `"no-chapter"` (`"Bad Request"`), `"no-content"` (`"Bad Gateway"`), `"concurrent"` (`"Conflict"`), and `"llm-stream"` (`"Bad Gateway"`); no new entries are required for the continue route. **However**, when `executeContinue()` raises `ChatError("no-content", "Latest chapter is empty; nothing to continue", 400)`, the HTTP route SHALL respond with status 400 (carried by `ChatError.httpStatus`), which means the existing 400 status SHALL pair with the existing `"Bad Gateway"` title. Clients MUST treat the `detail` field as authoritative; the title is a coarse classification.
   - Unknown errors → HTTP 500 Problem Details (existing fallback).

The continue route SHALL NOT pass a `template` field to `executeContinue()`. The server-side prompt build SHALL always use the resolved `playground/_prompts/system.md` (or default `system.md`) — there is no per-request template override for the continue path. (Rationale: the Continue button is invoked from the reading flow, not from the prompt-editor preview flow, and the existing prompt-editor preview uses the dedicated `POST /api/prompt` endpoint rather than the chat route.)

#### Scenario: Successful continue via HTTP

- **GIVEN** a story whose latest chapter is non-empty with on-disk bytes `B0`
- **WHEN** the client sends `POST /api/stories/s1/n1/chat/continue` with auth headers (and any body, including empty), and the upstream LLM streams continuation bytes that, after server-side transforms, are appended to the chapter as `B1`
- **THEN** the server SHALL stream the LLM continuation, append each delta to the latest chapter file, AND respond HTTP 200 with body `{ chapter: <n>, content: C, usage: <TokenUsageRecord | null> }` where `C` is byte-for-byte equal to the chapter file contents read from disk after the stream completes (i.e. `C === B0 + B1`)

#### Scenario: Continue refused with no chapters

- **GIVEN** a story directory with zero chapter files
- **WHEN** the client sends `POST /chat/continue`
- **THEN** the server SHALL respond HTTP 400 with Problem Details `{ title: "Bad Request", status: 400, detail: "Cannot continue: no existing chapter file", type: <RFC 9457 type uri> }`, AND SHALL NOT modify any file on disk

#### Scenario: Continue refused with empty latest chapter

- **GIVEN** a story whose latest chapter file is whitespace-only
- **WHEN** the client sends `POST /chat/continue`
- **THEN** the server SHALL respond HTTP 400 with Problem Details whose `detail` is `"Latest chapter is empty; nothing to continue"`, AND SHALL NOT open the chapter file for writing

#### Scenario: Continue refused while another generation is active

- **GIVEN** the per-story generation lock for `(series, story)` is held by another in-flight `executeChat()` or `executeContinue()`
- **WHEN** the client sends `POST /chat/continue`
- **THEN** the server SHALL respond HTTP 409 with Problem Details whose `detail` is `"Another generation is already in progress for this story"`

#### Scenario: HTTP client disconnect aborts continue

- **GIVEN** an in-flight `POST /chat/continue` whose upstream LLM stream has emitted two deltas
- **WHEN** the HTTP client disconnects (TCP close)
- **THEN** `c.req.raw.signal` SHALL be aborted by the runtime, `executeContinue()` SHALL throw `ChatAbortError`, the chapter file SHALL retain the two deltas already appended, AND no HTTP response body SHALL be observable by the (now-disconnected) client (the server SHALL still log the abort)

#### Scenario: Mid-stream provider error returns 502

- **GIVEN** an in-flight `POST /chat/continue` whose upstream sends an SSE `error` chunk
- **WHEN** the SSE parser detects the error chunk
- **THEN** the server SHALL respond HTTP 502 with Problem Details `{ title: "Bad Gateway", status: 502, detail: "<provider message>" }`, AND the chapter file SHALL retain the bytes appended before the error chunk

#### Scenario: Path traversal on continue is rejected

- **GIVEN** a request `POST /api/stories/..%2F..%2Fetc/chat/continue` (or any series / name that fails `validateParams`)
- **WHEN** the route receives the request
- **THEN** the existing `validateParams` middleware SHALL reject with HTTP 400 BEFORE `executeContinue()` is invoked, AND SHALL NOT touch the filesystem

#### Scenario: Rate limiter applies to continue

- **GIVEN** the per-passphrase rate limiter is configured and the current passphrase has exceeded its window quota for chat operations
- **WHEN** the client sends `POST /chat/continue`
- **THEN** the rate limiter middleware SHALL respond HTTP 429 BEFORE `executeContinue()` is invoked, identical to its behaviour on `POST /chat`
