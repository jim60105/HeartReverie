## 2. Prompt construction for continue

- [x] 2.1 In `writer/lib/story.ts` add `parseChapterForContinue(rawContent: string): { userMessageText: string; assistantPrefill: string }` that:
  - Locates the **first** `<user_message>ÔÇŽ</user_message>` block via a non-greedy, **case-insensitive** regex (`/<user_message>([\s\S]*?)<\/user_message>/i`) ÔÇö case-insensitive matches the existing `stripPromptTags()` behaviour so manually-edited tag variants like `<USER_MESSAGE>` are still recognised.
  - When the block is found, sets `userMessageText` to the inner content (trimmed) and `assistantPrefill` to the chapter bytes with that match removed, then run through `stripPromptTags()` and trimmed.
  - When the block is absent, sets `userMessageText = ""` and `assistantPrefill = stripPromptTags(rawContent).trim()`.
  - Documented behaviour for multiple `<user_message>` blocks: only the first is extracted; subsequent ones are removed by `stripPromptTags()` from the prefill.
  - Documented behaviour for literal `</user_message>` inside a user-message body: the regex stops at the first closing tag, so the remainder leaks into `assistantPrefill`. This is accepted as "manual editing of the user_message body to include a literal closing tag is unsupported"; a test SHALL pin this current behaviour so any future change is intentional.
- [x] 2.2 In `writer/lib/story.ts` add `buildContinuePromptFromStory(storyDir, systemTemplate)` that:
  - Reads all chapter files for the story in numeric order (re-loading from disk on every call ÔÇö no caching).
  - Errors with `ChatError("no-chapter", "Cannot continue: no existing chapter file", 400)` when there are zero chapter files.
  - **Re-reads the latest chapter file fresh from disk** via `Deno.readTextFile(chapterPath)` to honour any manual user edits made since the previous invocation. Calls `parseChapterForContinue()` on the freshly-read bytes.
  - Errors with `ChatError("no-content", "Latest chapter is empty; nothing to continue", 400)` when **both** `userMessageText.trim() === ""` *and* `assistantPrefill.trim() === ""`. (Either part being non-empty is sufficient to allow continue.)
  - Calls `renderSystemPrompt(...)` with `userInput: userMessageText`, `isFirstRound: false`, and the existing `previousContext` array built from chapters 1..nÔłĺ1 (same shape as `buildPromptFromStory()` produces, including plugin `prompt-assembly` hook dispatch).
  - Returns `{ messages, targetChapterNumber, existingContent, userMessageText, assistantPrefill }` where `messages` is the rendered array; **when `assistantPrefill.trim() !== ""`** one extra trailing entry `{ role: "assistant", content: assistantPrefill }` is appended after `renderSystemPrompt()` returns, and **when `assistantPrefill.trim() === ""`** the rendered array is returned without modification (no empty-content assistant message is sent ÔÇö the request is equivalent to a `chat:send` carrying `userMessageText`). `existingContent` is the **unstripped** chapter-n bytes captured at parse time and is also returned to the persistence layer for the snapshot check (task 1.2).
- [x] 2.3 Add unit tests `writer/lib/story_test.ts` (or extend the existing one) covering: zero chapters Ôćĺ throws `no-chapter`; chapter with only `<user_message>` and no prose Ôćĺ `userMessageText` is non-empty, `assistantPrefill` is `""`, allowed; chapter with prose only and no `<user_message>` Ôćĺ `userMessageText` is `""`, `assistantPrefill` is non-empty, allowed; chapter with both empty Ôćĺ throws `no-content`; chapter with edited `<user_message>` (different from the one originally sent via `chat:send`) Ôćĺ the parsed `userMessageText` reflects the edit; chapter with edited prose tail Ôćĺ the parsed `assistantPrefill` reflects the edit; chapter with two `<user_message>` blocks Ôćĺ only the first is extracted, the second is stripped by `stripPromptTags()`; chapter with `<USER_MESSAGE>` (uppercase) Ôćĺ still recognised due to the case-insensitive regex; chapter with a literal `</user_message>` inside the user-message body Ôćĺ the regex stops at the first close tag and the remainder leaks into `assistantPrefill` (pin this behaviour so future changes are intentional).
