## MODIFIED Requirements

### Requirement: lore_all Variable Generation
The lore system SHALL provide a `lore_all` template variable that contains all in-scope, enabled lore passages concatenated with separator strings. Each passage's content SHALL be rendered through the Vento template engine before concatenation using a two-pass approach: the first pass collects all passage bodies and generates an initial set of lore variable names and raw concatenated values (used as the rendering context snapshot); the second pass renders each **individual passage body** through Vento using an **immutable snapshot** of the first-pass lore variables plus core variables (such as `series_name` and `story_name`). After rendering, lore variables (including `lore_all`) are re-generated by concatenating the individually-rendered passage bodies. Passages that fail to render SHALL fall back to their raw content.

#### Scenario: Multiple passages concatenated with rendered content
- **WHEN** there are 3 enabled passages with priorities [100, 80, 60] and content ["Alice bio", "World of {{ series_name }}", "Plot setup"] and the current series is "fantasy"
- **THEN** `lore_all` SHALL be "Alice bio\n\n---\n\nWorld of fantasy\n\n---\n\nPlot setup" (sorted by priority descending, Vento syntax resolved)

#### Scenario: Single passage available
- **WHEN** there is 1 enabled passage with content "Only character description"
- **THEN** `lore_all` SHALL be "Only character description" (no separators)

#### Scenario: No passages in scope
- **WHEN** no lore passages are enabled or in-scope for the current story
- **THEN** `lore_all` SHALL be an empty string ""

#### Scenario: Passage with rendering error falls back to raw content
- **WHEN** a passage contains invalid Vento syntax `{{ broken( }}` alongside a valid passage with content "World rules"
- **THEN** `lore_all` SHALL contain the raw content of the broken passage (with invalid syntax preserved) concatenated with "World rules"

### Requirement: Tag-Specific Variable Generation
The lore system SHALL provide template variables for each unique effective tag, with tag names **normalized** to valid identifiers: lowercased, hyphens and spaces replaced with underscores, non-alphanumeric/underscore characters removed. The variable name format is `lore_<normalized_tag>`. If two tags normalize to the same identifier, their passages are merged under that variable. Each passage's content SHALL be rendered through the Vento template engine before concatenation using the same two-pass approach as `lore_all`: the first pass generates raw lore variables from unrendered passage bodies; the second pass renders each **individual passage body** through Vento using an immutable snapshot of the first-pass context; tag-specific variables are then re-generated by concatenating the rendered passage bodies.

#### Scenario: Character and world tags with Vento rendering
- **WHEN** there are passages with effective tags ["character", "world"] and content ["Alice bio", "The world of {{ series_name }}"] and the current series is "fantasy"
- **THEN** template variables `lore_character` SHALL be "Alice bio" and `lore_world` SHALL be "The world of fantasy"

#### Scenario: Hyphenated tag normalization
- **WHEN** passages have effective tag "comic-relief" with content ["Funny NPC"]
- **THEN** the template variable SHALL be `lore_comic_relief` (hyphen normalized to underscore)

#### Scenario: Multiple passages per tag
- **WHEN** there are 2 passages both tagged "character" with priorities [90, 70] and content ["Alice", "Bob"]
- **THEN** `lore_character` SHALL be "Alice\n\n---\n\nBob" (sorted by priority descending)

#### Scenario: Inter-lore reference in tag-specific variable
- **WHEN** a passage tagged "scenario" contains `{{ lore_character }}` and a passage tagged "character" contains "Alice the Brave"
- **THEN** `lore_scenario` SHALL contain "Alice the Brave" in place of the `{{ lore_character }}` reference after the second rendering pass

### Requirement: Prompt Integration
The lore retrieval engine in `writer/lib/lore.ts` SHALL be called directly by `renderSystemPrompt()` in `writer/lib/template.ts` with the active series and story context. The engine SHALL use a two-pass approach with an immutable snapshot: first, collect all passage bodies and compute raw lore template variables (variable names and raw concatenated values); second, render each **individual passage body** through the Vento template engine using an **immutable snapshot** of all first-pass lore variables plus core variables (`series_name`, `story_name`, and other template variables available at render time) — the snapshot SHALL NOT be modified during iteration. After rendering, lore variables are re-generated from the individually-rendered passage bodies. The final lore variables SHALL then be returned to be spread into the Vento template render context alongside existing variables.

#### Scenario: Variables computed at render time with immutable-snapshot two-pass rendering
- **WHEN** the system prompt is being rendered via `renderSystemPrompt()` with series "fantasy" and story "quest"
- **THEN** the lore retrieval engine SHALL first compute raw lore variables (the immutable snapshot), then render each individual passage body through Vento against that snapshot, re-generate lore variables from the rendered bodies, and make the final lore variables available in the Vento template context

#### Scenario: Variables integrated with existing variables
- **WHEN** the template is rendered with both existing template variables and lore variables
- **THEN** lore variables SHALL be merged with other template variables without conflicts

#### Scenario: Core variables accessible during lore rendering pass
- **WHEN** a lore passage references `{{ series_name }}` and the current series is "fantasy"
- **THEN** the second rendering pass SHALL resolve `{{ series_name }}` to "fantasy" because core variables are included in the immutable snapshot alongside the first-pass lore variables
