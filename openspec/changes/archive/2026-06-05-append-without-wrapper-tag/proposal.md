## Why

The plugin-action append path (`runPluginPrompt(file, { append: true, appendTag })`) was designed for plugins whose LLM output is a single logical block that the *engine* should wrap in one canonical XML envelope (e.g. `state` wrapping its output in `<UpdateVariable>`). But the `sd-webui-image-gen` plugin's "設計新圖片" button uses a prompt that itself emits **multiple** `<image>` blocks interleaved with narrative prose. Because `append: true` mandates an `appendTag` and the engine always wraps the *entire* accumulated response in exactly one `<{appendTag}>…</{appendTag}>` layer, that plugin's output gets a spurious outer `<image>` wrapping a payload that already contains several `<image>` blocks plus prose — producing malformed, nested `<image>` markup the tag was never meant to express. The append mode currently has **no way** to say "append the model's output verbatim, the prompt already owns its own tag structure."

## What Changes

- Make `appendTag` **optional** on the `append: true` plugin-action path. When a request sets `append: true` and omits `appendTag` (i.e. it is `undefined`/absent), the route SHALL append the normalised model output to the highest-numbered chapter **without adding any wrapper element** — appending `\n{normalised content}\n` instead of `\n<{appendTag}>\n{normalised content}\n</{appendTag}>\n`.
- When `appendTag` IS supplied (a non-empty string), behaviour is unchanged: the existing single-outer-wrapper normalisation strips at most one matching outer `<{appendTag}>` layer and re-wraps in exactly one `<{appendTag}>` envelope.
- In the tagless append case, the `post-response` hook payload's `appendedTag` SHALL be `null`, and the final result / `plugin-action:done` envelope SHALL report `chapterUpdated: true`, `appendedTag: null`.
- **BREAKING (type-only, no runtime consumer impact expected):** widen the `post-response` hook payload field `PostResponsePayload.appendedTag` from `?: string` to `?: string | null`. Today the append finalizer is the ONLY site that sets it (always a non-null string) and replace/write-new/discard omit it; the tagless path now sets it to `null`, which does not type-check against `?: string` under `strictNullChecks`. The sibling result/envelope/frontend types (`PluginRunPromptResponse.appendedTag`, the WS `plugin-action:done` `appendedTag`, the frontend option/result types) are ALREADY `string | null`, so only the hook payload type and its spec text (`plugin-hooks`) need to change.
- **No-tag normalisation:** in the tagless case the route SHALL still `trim()` the accumulated content before appending, but SHALL NOT attempt any wrapper stripping (there is no tag to match against).
- Tighten the existing validation so the combinations stay coherent and symmetric across modes: `append: true` + `appendTag` present-but-invalid (fails the `^[a-zA-Z][a-zA-Z0-9_-]{0,30}$` regex, or is any non-string value INCLUDING explicit `null`) is STILL rejected with `plugin-action:invalid-append-tag`; `replace: true` + a non-`undefined` `appendTag` (including `null`) is STILL rejected with `plugin-action:invalid-replace-combo`. ONLY a totally-omitted (`undefined`) `appendTag` on the append path opts into tagless append. Treating explicit `null` and `undefined` identically is deliberately avoided so a hand-written client gets consistent behaviour across append/replace (omission is the single, unambiguous opt-in).
- The frontend `runPluginPrompt` helper already forwards `appendTag` only when defined and already types it optional — no behavioural change is required there beyond confirming the contract and adding a regression test for the omitted-tag path.

This is **not** a behavioural breaking change: every existing caller passes an `appendTag` and keeps its current runtime behaviour. The only newly-reachable runtime state is "append + omitted tag", which previously returned HTTP 400. There is one type-only widening (`PostResponsePayload.appendedTag` gains `| null`) that well-typed in-repo `post-response` consumers already tolerate (the field is optional today and `null` for non-append modes is the natural extension); it is called out as BREAKING above only because it changes a public type.

## Capabilities

### New Capabilities
_None._ This extends the existing plugin-action run-prompt contract; no new top-level capability is introduced.

### Modified Capabilities
- `plugin-action-buttons`: Relax the run-prompt route so `appendTag` is optional (opt-in via total omission) in append mode; define the tagless append persistence behaviour (no wrapper element, trim-only normalisation, `appendedTag: null` in the response/`plugin-action:done` payload). Confirm the `runPluginPrompt` helper contract for the omitted-`appendTag` path.
- `plugin-hooks`: Widen the `post-response` payload's `appendedTag` field to `string | null` (was `?: string`) so the tagless append can report `appendedTag: null`. Document that `appendedTag` is `null` for a tagless plugin-action append (in addition to being omitted for chat / replace / write-new).

## Impact

- **Backend validation**: `writer/routes/plugin-actions-validation.ts` (`validateModeCombo` — accept absent `appendTag` in append mode, keep rejecting present-but-invalid tags), `writer/routes/plugin-actions.ts` / `plugin-actions-preflight.ts` (thread the now-nullable `appendTag` through), `writer/lib/errors.ts` (no new slug needed).
- **Backend persistence**: `writer/lib/chat-types.ts` (`WriteMode` `append-to-existing-chapter` variant — make `appendTag` `string | null`), `writer/lib/chat-chapter-finalize.ts` (`finalizeAppendToExisting` — branch on `appendTag === null` to append without wrapper; `normaliseAppendContent` stays a helper used only when a tag is present), `writer/routes/plugin-actions-execute.ts` (pass `appendTag: validatedAppendTag` which may be `null`).
- **Hook payload type**: `writer/types/hooks.ts` (`PostResponsePayload.appendedTag` `?: string` → `?: string | null`, JSDoc updated to note `null` for tagless append). Confirm the four other `post-response` dispatch sites (`finalizeWriteNewChapter`, `finalizeReplace`, `finalizeContinueLastChapter`, `discard`) still satisfy the widened-and-still-optional type — they omit the field, which `?:` permits.
- **Types / WS**: `writer/types/plugin.ts` (`appendedTag: string | null` — unchanged), `writer/types/ws.ts` (`appendedTag: string | null` — unchanged), `writer/routes/ws-plugin-action.ts` (already passes `msg.appendTag` straight through — confirm `undefined` survives and that the client envelope omits the key), `writer/routes/plugin-actions-shared.ts`.
- **Frontend**: `reader-src/src/composables/useChatApi.ts` and `reader-src/src/types/index.ts` — already optional; add a regression test asserting that `runPluginPrompt(file, { append: true })` with no `appendTag` sends a body/WS envelope WITHOUT an `appendTag` field.
- **Tests**: backend unit tests for tagless append (exact appended bytes `\n{trimmed}\n` with no wrapper element, `appendedTag: null` in `post-response`); route-level tests for the newly-valid "append + omitted tag" combination and the still-rejected "append + bad tag", "append + explicit null tag", and "replace + tag (incl. null)" combinations; frontend tests for the omitted-tag forwarding on BOTH the WebSocket envelope and the HTTP body.
- **Docs**: `docs/plugin-dev/action-buttons.md`, `AGENTS.md` action-button summary, and the `heartreverie-create-plugin` skill (`SKILL.md` + `references/hook-api.md`) updated to document that `appendTag` is optional and what tagless append does.
- **Downstream**: unblocks the sibling `HeartReverie_Plugins` change that switches the `sd-webui-image-gen` "設計新圖片" button to `runPluginPrompt("image-design.md", { append: true })` (no tag). Per workspace sequencing, this core change lands first.
