## Why

Authors often want to take a freshly-generated chapter draft and rewrite it in a more literary, dialogue-driven Chinese prose style without re-running the full chat round (which would also recompute upstream context, lore retrieval, and the `previousContext` chain). HeartReverie has all the streaming, cancellation, token-usage, and hook plumbing it needs in `streamLlmAndPersist`, but no first-class "polish" affordance: the only chapter-mutating plugin-action mode today is `append-to-existing-chapter`, which adds a wrapped block instead of rewriting the prose itself. Adding a "潤飾" (Polish) round closes that gap with a single click.

## What Changes

- Add a new bundled plugin `polish` under `HeartReverie/plugins/polish/` that contributes one action button (`id: "polish"`, `visibleWhen: "last-chapter-backend"`) and ships a dedicated rewrite system prompt at `plugins/polish/polish-instruction.md`.
- Add a new `WriteMode` variant `{ kind: "replace-last-chapter" }` to `writer/lib/chat-shared.ts` that, on successful stream completion, atomically replaces the highest-numbered chapter file with the accumulated LLM output via the existing `atomicWriteChapter` helper (write-to-temp + `Deno.rename`). On abort or error the original chapter file SHALL remain byte-for-byte unchanged.
- Extend the `POST /api/plugins/:pluginName/run-prompt` route (and the matching WebSocket `plugin-action:run` handler) to accept a new request shape: `{ append: false, replace: true }` selects `replace-last-chapter` mode. `append` and `replace` SHALL be mutually exclusive; `replace: true` requires at least one existing chapter file (HTTP 400 `plugin-action:no-chapter` otherwise) and SHALL NOT accept `appendTag`.
- The polish prompt template SHALL receive the current last-chapter content as the Vento variable `draft` (string) sourced server-side from the chapter file. The server SHALL apply `PluginManager.getStripTagPatterns()` (the same `promptStripTags` regex used to scrub prompt history in `chat-shared.ts`) to the raw chapter content before binding it to `draft`, so control tags such as `<user_message>` and `<chapter_summary>` are removed before the model sees the draft. The template SHALL emit a `{{ message "user" }}…{{ /message }}` block that asks the LLM to rewrite the draft in literary modern Chinese prose with: dialogue-driven advancement, "show don't tell", smooth scene transitions, faithful character voices, full-width Chinese punctuation, no bullet-point lists. The prompt SHALL be SFW and shall not contain any 18+/jailbreak/age-related directives.
- The frontend SHALL surface the polish button through the existing `PluginActionBar` + `runPluginPrompt` plumbing with `{ replace: true }`. The button SHALL be disabled while `isLoading` is true, while `visibleWhen: "last-chapter-backend"` is unsatisfied (no chapter, not on last chapter, frontend-only mode), AND while the chapter editor in `ChapterContent.vue` has an unsaved buffer for the current chapter; the button SHALL preserve the original chapter content in the UI if the user clicks Stop mid-stream. On `chapterReplaced: true` the frontend SHALL force-close the chapter editor for the affected chapter, drop its stale `editBuffer`, and reload the chapter content so a subsequent save can never overwrite the polished file with the pre-polish draft.
- Token usage for polish rounds SHALL be persisted via the existing `appendUsage` path using the existing `TokenUsageRecord` schema unchanged (`{ chapter, promptTokens, completionTokens, totalTokens, model, timestamp }`). Polish rounds SHALL NOT introduce new fields on the record — the v1 usage panel does not distinguish chat-driven and plugin-action-driven usage and no schema migration is in scope.
- The `replace` request shape SHALL inherit the existing per-route 30-rpm rate limiter, the per-story generation lock, the path-traversal-safe prompt resolution, and the `extraVariables` reserved-name guard already enforced by `runPluginActionWithDeps`. The reserved-name list SHALL grow to include `draft` so callers cannot override the server-supplied draft via `extraVariables`.

## Capabilities

### New Capabilities
_None._ The polish plugin is implemented entirely on top of existing capabilities; no new top-level spec is created.

### Modified Capabilities
- `plugin-action-buttons`: Add `replace: true` (and the `replace-last-chapter` `WriteMode`) to the run-prompt route and `runPluginPrompt` helper. Document `replace`/`append` mutual exclusion, the `draft` reserved variable, the atomic-replace persistence behaviour, the cancel/error rollback guarantee, and the `polish` bundled plugin shipping a `polish` action button.

## Impact

- **New plugin directory**: `plugins/polish/` (`plugin.json`, `polish-instruction.md`, `README.md`).
- **Backend types**: `writer/lib/chat-shared.ts` (`WriteMode` union, `streamLlmAndPersist` finalisation branch); `writer/routes/plugin-actions.ts` (`replace` body field validation, `WriteMode` selection, `RESERVED_VARIABLE_NAMES` += `draft`, server-side load of last-chapter content into `extraVariables.draft`, `pluginActionProblems.invalidReplaceCombo` / `noChapter`); `writer/routes/ws.ts` (mirror the new field on the WebSocket envelope); `writer/types.ts` (`PluginRunPromptRequest`, `PluginRunPromptResponse.chapterReplaced`).
- **Frontend**: `reader-src/src/types/index.ts` (`RunPluginPromptOptions.replace`); `reader-src/src/composables/useChatApi.ts` (forward `replace` over WS + HTTP); `reader-src/src/components/ChapterContent.vue` and/or `reader-src/src/composables/useChapterActions.ts` (expose an "editor has unsaved buffer for chapter N" predicate to gate the polish button, and a "force-close editor + reload chapter N" hook that fires on `chapterReplaced: true`); `reader-src/src/composables/usePluginActions.ts` (consult the editor predicate when computing the polish button's disabled state). No new component — the polish button reuses `PluginActionBar`.
- **Tests**: backend unit tests for the new `WriteMode` (atomic replace on success, no chapter mutation on abort/error), route-level tests for `replace`/`append` mutual exclusion and the `draft` reserved-name collision; a strip-tags test verifying that a chapter file containing `<user_message>...</user_message>` and `<chapter_summary>...</chapter_summary>` is bound to the Vento `draft` variable with both wrappers removed; frontend tests for the new option being forwarded; a frontend test confirming the polish button is disabled while the editor has an unsaved buffer for the current chapter and that `chapterReplaced: true` force-closes the editor and reloads the chapter; an integration test driving the end-to-end polish round.
- **No breaking changes**: the new mode is opt-in and the existing `append`/`discard` paths are untouched.
