## Why

When a user reloads the browser at a chapter URL (e.g. `https://localhost:8443/<series>/<story>/chapter/1`) or saves an in-place edit via the chapter toolbar, plugin-driven frontend rendering (e.g. the third-party `status` plugin's `<status>…</status>` panels) is not consistently triggered. The exact same content renders correctly when the user reaches the chapter via the story selector or the next/previous chapter controls. Chapters render as raw, partially-stripped XML or with a missing sidebar panel, breaking the reading experience for stories that depend on plugin renderers.

Three distinct ordering bugs are at play, ranked from most-confirmed to least-confirmed:

1. **Stale sidebar relocation effect (high confidence — directly explains the `status` plugin symptom).** The `status` plugin renders its panel as `<div class="status-panel main-card plugin-sidebar">…</div>` *inside* a chapter's HTML token. `ContentArea.vue`'s `watchPostEffect` then relocates every `.plugin-sidebar` element from the content into the `<Sidebar>` element. The effect tracks **only** `currentContent.value` and `isLastChapter.value`. It does NOT track plugin readiness or any "render invalidation" signal, so:
   - On reload, if `<ChapterContent>` happens to mount and render once before plugins are registered, the sidebar relocation runs against content that contains no `.plugin-sidebar` (because the `<status>` block was stripped by DOMPurify). When plugins eventually become ready and the next render produces a `.plugin-sidebar`, the relocation effect's dependencies haven't changed (same `currentContent` string), so it never re-runs and the panel is left orphaned inside the content area or stripped from the DOM.
   - On edit-save, if the new content is byte-identical to the old content (or if `reloadToLast` jumps to the last chapter and that chapter's content is unchanged), the relocation effect doesn't re-run — even if `tokens` was forced to re-evaluate — so the sidebar stays empty or stale.

2. **Edit-save jumps to the wrong chapter and may not invalidate rendering.** After `editChapter()` succeeds, `ChapterContent.vue` calls `useChapterNav().reloadToLast()`. This always **jumps to the last chapter** — even when the user edited an earlier chapter — so the user sees a different chapter than the one they just edited (a separate UX bug that has been there since the feature shipped). It also unconditionally clobbers `currentContent`, but if the edited content is byte-identical to what was already in memory the `tokens` computed never re-evaluates because Vue's primitive ref reactivity uses `Object.is` equality. The `chapter:render:after` hook never re-fires for the edited chapter, so plugins that mutate tokens silently lose state.

3. **Possible reload race (lower confidence — needs instrumentation before refactoring).** On a deep-link reload, `App.vue#handleUnlocked` does `await Promise.all([initPlugins(), applyBackground()])` *then* `await loadFromBackend(...)`, so in principle plugins are ready before `currentContent` is set. However, several conditions could still cause `renderChapter()` to dispatch hooks against an empty handler set: (a) plugin `register()` functions that are themselves async are not currently `await`ed by `usePlugins`, so a plugin can resolve its dynamic import without having finished registering; (b) `usePlugins.initPlugins()` has no in-flight-promise idempotency, so two concurrent callers can race; (c) a transient `/api/plugins` failure silently produces a "no plugins" experience that this change must surface, not hide. Before refactoring deep-link load ownership, we will add instrumentation to confirm whether and when this race actually fires in the wild.

This change closes the confirmed ordering gaps (1 and 2) and adds the readiness gate plus instrumentation needed to confirm or rule out (3) before further refactoring. Plugin-driven rendering becomes **deterministic** in every navigation path: story selector, next/previous, reload-at-chapter-URL, edit-and-save, rewind, and branch.

## What Changes

- **Sidebar relocation effect tracks render invalidation.** `ContentArea.vue`'s `watchPostEffect` is replaced with an explicit `watch([currentContent, isLastChapter, pluginsReady, renderEpoch], …, { flush: "post", immediate: true })` that re-runs whenever any of those signals changes. The effect SHALL also clear the sidebar whenever it runs against a "not yet renderable" state (no current content, or plugins not ready) so stale panels from a previous chapter never leak.
- **Plugin readiness gate before first render.** `usePlugins()` exposes a reactive `pluginsReady` ref (true once `initPlugins()` resolves) **and** a `pluginsSettled` ref (true once `initPlugins()` has run to completion, success or failure). The two are decoupled so plugin-load failures cannot silently masquerade as "no plugins". `ContentArea.vue` gates `<ChapterContent>` rendering on `pluginsSettled && currentContent`, falling back to a minimal loading placeholder. `pluginsReady` is exposed for diagnostics and as a dependency of the sidebar relocation effect; failure to load plugins results in a single visible warning toast and a logged error, not a permanently hidden chapter.
- **Idempotent, async-safe plugin initialization.** `initPlugins()` is rewritten to use a module-level in-flight `Promise<void>` so concurrent callers share one initialization. Plugin `register(hooks)` calls are `await Promise.resolve(register(...))` so async `register` functions are honored.
- **Force a re-render on chapter content mutation.** `currentContent` is upgraded to a `shallowRef<string>` and `useChapterNav()` exposes a private `commitContent(next)` helper that calls `triggerRef(currentContent)` whenever the new value is `===` to the old value, so byte-identical reloads still invalidate downstream computeds and effects. A separate module-level `renderEpoch: Ref<number>` is bumped in lockstep so non-`shallowRef`-aware consumers (the sidebar relocation watch) can also observe the invalidation.
- **`reloadToLast` is split by intent.** `refreshAfterEdit(targetChapter)` is added — reloads the chapter list, **stays on the chapter the user edited** (not the last chapter), and forces a content invalidation even when the on-disk text is byte-identical. `reloadToLast()` is preserved unchanged for genuine "go to the new last chapter" callers (rewind, branch, post-LLM-stream).
- **Re-render contract on plugin readiness change.** `ChapterContent.vue` reads `pluginsReady` and `renderEpoch` inside its `tokens` computed so a render that occurred before plugins were ready (or before the latest invalidation) is corrected once those signals change.
- **Diagnostic instrumentation for the reload path.** A `RENDER_DEBUG` env-gated console log channel records the ordering of `auth verified`, `plugins settled`, `loadFromBackend committed`, and `renderChapter dispatched (handler count: N)` events. The implementation phase will use this to confirm or rule out the residual reload race **before** any deeper refactor of deep-link load ownership.
- **Conditional deep-link refactor (Phase 2).** **Only if** instrumentation confirms a reload race that the gate alone does not close, App.vue's deep-link `loadFromBackend()` call is replaced with an `immediate`, plugin-readiness-gated route watcher inside `useChapterNav.initRouteSync()` that owns deep-link loads through a "desired route key" model (immune to mid-load route changes). Otherwise, `App.vue` retains its current ownership.
- **Test coverage closing both confirmed regressions.** New Vitest suites assert: (a) `chapter:render:after` invalidates exactly when the edit-save flow commits content (including byte-identical commits), (b) the sidebar relocation effect re-runs and a `.plugin-sidebar` produced after an edit-save correctly ends up inside the `<Sidebar>` element, (c) editing chapter N keeps the user on chapter N (not the last chapter), (d) `pluginsSettled` flips on both success and failure paths, and (e) story-selector and next-chapter paths remain unchanged.
- **Documentation.** `docs/plugin-system.md` adds a "Render lifecycle" section describing the readiness gate, the sidebar relocation contract, and the navigation paths that trigger `frontend-render` / `chapter:render:after`.

There is **no breaking change** for plugin authors: hook signatures, manifest fields, and registration order are unchanged. The fix is internal to the reader composables and `App.vue`. There is no migration step because, per the user's note, the project has zero deployed users.

### Modified Capabilities
- `plugin-hooks`: tighten the rendering-time contract — the reader MUST NOT dispatch `frontend-render` or `chapter:render:after` for any chapter until plugin initialization has settled; introduce reactive `pluginsReady` (success) and `pluginsSettled` (success-or-failure) flags; require idempotent in-flight `initPlugins`; require `await Promise.resolve(register(...))`.
- `vue-component-architecture`: `ContentArea` gates `<ChapterContent>` on `pluginsSettled && currentContent` and uses an explicit `watch([currentContent, isLastChapter, pluginsReady, renderEpoch], …, { flush: "post" })` for sidebar relocation that clears stale panels when not renderable.
- `chapter-editing`: replace the existing "Edit flow updates content without reload" requirement so edit-save (a) keeps the user on the chapter they just edited, and (b) guarantees the markdown renderer re-runs and `chapter:render:after` re-fires even when the new content is byte-identical to the old content.
- `chapter-navigation`: clarify that `loadFromBackend()`, `reloadToLast()`, and the new `refreshAfterEdit()` are the only public entry points that mutate `currentContent` for backend mode, and that all three SHALL invalidate the rendered chapter view (using `shallowRef`+`triggerRef` and a `renderEpoch` ref) regardless of byte-equality.
- `md-renderer`: add a requirement that `renderChapter()` MUST be invoked only after `pluginsSettled` is `true`, and that token re-evaluation is forced when content is invalidated even if the input string is `===` to the previous input.

## Impact

- **Affected source**:
  - `reader-src/src/composables/usePlugins.ts` — add `pluginsReady` and `pluginsSettled` refs; module-level in-flight `initPromise` for idempotency; `await Promise.resolve(register(...))`; surface plugin-load errors via the existing notification toast composable; never silently fall back to "no plugins" without a visible signal.
  - `reader-src/src/composables/useChapterNav.ts` — convert `currentContent` to `shallowRef<string>`; introduce a private `commitContent(next)` that calls `triggerRef` on identical writes; expose a module-level `renderEpoch: Ref<number>` bumped in lockstep; route every content write (`loadFromBackend`, `reloadToLast`, `pollBackend`, WebSocket `chapters:content`, the new `refreshAfterEdit`) through `commitContent`; add `refreshAfterEdit(targetChapter)`.
  - `reader-src/src/components/ContentArea.vue` — replace `watchPostEffect` with `watch([currentContent, isLastChapter, pluginsReady, renderEpoch], …, { flush: "post", immediate: true })`; clear sidebar when not renderable; gate `<ChapterContent>` on `pluginsSettled && currentContent`.
  - `reader-src/src/components/ChapterContent.vue` — read `pluginsReady` and `renderEpoch` inside `tokens` computed; call `refreshAfterEdit(currentChapterNumber)` instead of `reloadToLast()` after a successful save.
  - `reader-src/src/App.vue` — only refactored if Phase 2 instrumentation confirms a residual reload race; otherwise unchanged. The `pluginsSettled` gate alone may be sufficient.
  - `reader-src/src/lib/plugin-hooks.ts` — add `getHandlerCount(stage: HookStage): number` so instrumentation and tests can verify hook registry population.
- **Affected tests**:
  - `reader-src/src/components/__tests__/ChapterContent.test.ts` — add edit-save re-render scenarios, byte-identical edit scenarios, sidebar relocation scenarios.
  - `reader-src/src/components/__tests__/ContentArea.test.ts` (new or extended) — assert `<ChapterContent>` does not mount before `pluginsSettled`, and that the sidebar relocation watch clears stale panels when content/plugins change.
  - `reader-src/src/composables/__tests__/useChapterNav-coverage.test.ts` — assert `refreshAfterEdit` keeps current index on the edited chapter, byte-identical commits trigger `triggerRef` and bump `renderEpoch`.
  - `reader-src/src/composables/__tests__/usePlugins-pluginsReady.test.ts` (new) — assert `pluginsReady` is `true` only on full success; `pluginsSettled` is `true` after both success and failure paths; concurrent `initPlugins()` calls share one `initPromise`; async `register()` is awaited.
- **No backend changes**: backend chapter-edit/branch/rewind endpoints, WebSocket protocol, and plugin manifest schema are untouched.
- **Documentation**: `docs/plugin-system.md` gains a "Render lifecycle" section; `AGENTS.md`'s Frontend Rendering Pipeline blurb gets a one-paragraph update describing the readiness gate, sidebar relocation contract, and edit-save invariant.
